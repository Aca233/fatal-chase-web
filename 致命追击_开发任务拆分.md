# 《致命追击：混沌法则》开发任务拆分

## 1. 任务目标

本任务拆分用于把 GDD、架构计划和 UI/UX 方案转化为可执行 backlog。当前阶段建议先完成规则细化和 M1 网页端本地原型，再进入权威联网。

首个可玩目标：

- 浏览器中运行。
- 2 到 4 个本地玩家或 AI 假人实体。
- WASD 移动、鼠标瞄准、蓄力射箭、取消假动作、翻滚。
- 追击链目标指引和目标转移。
- 基础断肢实战影响。
- 木箱、草丛、泥沼。
- 基础 HUD、肢体状态 UI、结算页。

## 2. 执行阶段总览

```text
P0 规则与原型准备
├─ 明确核心规则边界

├─ PixiJS/Matter.js 场景

├─ 断肢状态与追击链
```

## 3. P0：规则与原型准备

### T0.1 建立规则细化文档

优先级：P0  
依赖：现有 GDD、架构计划  
产出：

- `追击链规则说明.md`
- `肢体损失规则说明.md`
- `亡者系统状态机.md`

验收：

- 覆盖第三方击杀、环境击杀、同 tick 多人死亡、只剩 1 到 2 人的追击链收敛。
- 覆盖断臂、断腿、双臂断、双腿断、躯干重伤、头部命中的玩法效果。
- 覆盖寄生、夺舍、幽灵显形、假射线冷却和反制窗口。

### T0.2 建立 Web 项目骨架

优先级：P0  
依赖：T0.1 可并行  
建议技术：

- TypeScript
- Vite
- PixiJS
- Matter.js
- Node.js
- WebSocket 预留

目录目标：

```text
FatalChaseWeb/
├─ apps/client
├─ apps/server
├─ packages/shared
├─ packages/config-schema
├─ configs
├─ docs
└─ tests
```

验收：

- `apps/client` 能启动空白 PixiJS 场景。
- `apps/server` 能启动健康检查接口或最小 WebSocket 服务。
- `packages/shared` 能被 client 和 server 引用。
- TypeScript、lint、format 基础脚本可运行。

### T0.3 制定 UI token 与 HUD 线框

优先级：P0  
依赖：UI/UX 方案  
产出：

- `视觉_token草案.md`
- `局内HUD线框图.md`
- `核心组件状态表.md`

验收：

- 明确颜色、字体、描边、层级、危险提示、动效时长。
- HUD 覆盖生者、断肢、亡者、毒圈、大招阶段。
- 1366x768 下关键 HUD 不重叠。

### T0.4 建立配置草案

优先级：P1  
依赖：T0.1  
产出：

- `configs/characters.json`
- `configs/weapons.json`
- `configs/limbs.json`
- `configs/maps.json`
- `configs/match_phases.json`
- `configs/awards.json`

验收：

- 移动、翻滚、蓄力、断肢倍率、幽灵显形、草丛隐身等基础数值可配置。
- 配置字段与 `packages/shared` 类型一致。

## 4. P1：M1 本地战斗原型

### T1.1 实现场景与渲染基础

优先级：P0  
依赖：T0.2  
范围：

- PixiJS 游戏画布。
- 摄像机跟随。
- 地图边界。
- 简化火柴人/几何体角色。
- 调试图层。

验收：

- 浏览器中可看到地图、角色和基础调试信息。
- 画布随窗口尺寸变化保持比例正确。

### T1.2 实现输入系统

优先级：P0  
依赖：T1.1  
范围：

- WASD 移动。
- 鼠标瞄准。
- 左键蓄力。
- 右键取消蓄力。
- 空格翻滚。
- E 弹反预留。

验收：

- 输入被归一化为 `InputCommand`。
- 支持按键缓冲和冷却状态。
- 取消蓄力不会生成箭矢。

### T1.3 实现移动与翻滚

优先级：P0  
依赖：T1.2  
范围：

- 基础移动速度。
- 朝向。
- 翻滚位移。
- 0.5 秒无敌帧状态。
- 翻滚冷却 UI。

验收：

- 角色移动响应稳定。
- 翻滚期间状态可见，冷却结束可再次使用。
- 断腿倍率接入时不需要重写移动模块。

### T1.4 实现蓄力射箭与假动作

优先级：P0  
依赖：T1.2、T1.3  
范围：

- 蓄力曲线。
- 箭矢生成。
- 飞行速度。
- 命中检测。
- 右键假动作取消。
- 准星蓄力表现。

验收：

- 松开左键射出箭矢。
- 右键取消只播放动作/音效表现，不造成伤害。
- 箭矢能击中角色和木箱。

### T1.5 实现基础肢体状态

优先级：P0  
依赖：T1.4、T0.4  
范围：

- `LimbState` 类型。
- 断臂、断腿、双腿断裂。
- 能力倍率接入移动、蓄力和翻滚。
- 肢体状态 HUD。
- 角色模型部位缺失表现。

验收：

- 断臂后蓄力变慢或准星抖动。
- 断腿后移动变慢、翻滚距离变短。
- 双腿断后进入爬行且无法翻滚。
- HUD 能清楚显示缺失部位。

### T1.6 实现基础追击链

优先级：P0  
依赖：T1.1、T1.4  
范围：

- 2 到 4 个本地实体生成闭环目标。
- 当前目标指针。
- 击杀目标得分。
- 第三方击杀后的目标转移。
- 断链短反馈。

验收：

- 每个实体始终有唯一目标。
- 玩家不需要看说明也能知道追击方向。
- 抢怪后目标转移到抢怪者。

### T1.7 实现地图交互元素

优先级：P1  
依赖：T1.1、T1.4  
范围：

- 合金黑墙。
- 木箱掩体和破坏。
- 高草丛隐身。
- 泥沼减速。

验收：

- 黑墙阻挡移动和箭矢。
- 木箱可被箭矢破坏。
- 草丛内角色隐藏或降低可见性。
- 泥沼影响移动与翻滚。

### T1.8 实现基础 HUD

优先级：P0  
依赖：T0.3、T1.3、T1.5、T1.6  
范围：

- `GameHUD`
- `HuntCompass`
- `LimbStatus`
- `CooldownStrip`
- `ChargeReticle`
- `DangerPing`

验收：

- 关键 HUD 不遮挡角色和箭矢。
- 1366x768 下不重叠。
- 肢体状态、翻滚冷却、蓄力状态可读。

### T1.9 实现本地结算页

优先级：P1  
依赖：T1.5、T1.6  
范围：

- 胜者。
- 击杀分。
- 断肢统计。
- 假动作次数。
- 追击链变化摘要。
- `再来一局`。

验收：

- 单局结束后进入结算。
- 结算能展示荒诞称号。
- 可一键重开。

### T1.10 M1 原型验收

优先级：P0  
依赖：T1.1 到 T1.9  
验收清单：

- 浏览器可启动并完成一局本地原型。
- 玩家能移动、瞄准、射箭、取消、翻滚。
- 断肢实战影响可见且可感知。
- 追击链和目标转移可用。
- 地图至少包含木箱、草丛、泥沼。
- 局内 HUD 不遮挡核心危险。
- 结算页可重开。

## 5. P2：M2 亡者机制原型

### T2.1 死亡流程与形态选择

优先级：P0  
依赖：T1.10  
范围：

- 死亡后 0.3 秒反馈。
- 黄泉选择 UI。
- 寄生肉盾与幽灵老六两个入口。
- 3 秒默认选择策略。

验收：

- 死亡后不退出玩法。
- 玩家能在 3 秒内重新获得可操作形态。

### T2.2 寄生肉盾

优先级：P0  
依赖：T2.1  
范围：

- 绑定击杀者。
- 宿主挂点。
- 寄生 HUD。
- 宿主断肢状态影响寄生交互。

验收：

- 死者能挂在击杀者身上。
- 宿主移动时寄生者跟随。

### T2.3 空手接白刃与夺舍

优先级：P0  
依赖：T2.2、T1.4  
范围：

- 擦箭窗口。
- 接箭输入。
- 成功后宿主死亡。
- 寄生者复活。
- 肢体状态继承规则。

验收：

- 擦箭窗口足够短，有反应挑战。
- 成功夺舍后身份切换清晰。

### T2.4 幽灵老六

优先级：P0  
依赖：T2.1  
范围：

- 幽灵隐形。
- 假拉弓音效。
- 假红色射线。
- 显形 0.5 秒。
- 冷却。

验收：

- 幽灵能干扰生者。
- 生者能通过残影识别幽灵。
- 假射线不与真实箭矢预警混淆。

### T3.2 WebSocket 协议

## 9. 横向任务

### Q1 测试策略

优先级：持续  
范围：

- 规则服务单元测试。
- 追击链边界测试。
- 断肢倍率测试。
- 亡者状态机测试。
- UI 组件状态测试。
- Playwright 局内 HUD 截图检查。

验收：

- 每个核心规则模块有可重复测试。
- 关键 UI 在 1366x768 和 1920x1080 下不重叠。

### Q2 性能预算

优先级：持续  
范围：

- 60 FPS 目标。
- 粒子数量上限。
- 同屏喷涂上限。
- 箭矢和断肢残骸生命周期。
- 低配模式。

验收：

- M1 原型在普通桌面浏览器中稳定运行。
- 关闭血腥动效后仍保留核心战斗信息。

### Q3 可访问性

优先级：持续  
范围：

- 色盲可读。
- 降低屏幕震动。
- 减少血腥动效。
- 音频分类音量。
- 键位重绑定预留。

验收：

- 目标、威胁、幽灵、断肢不只靠颜色区分。
- 减少动效不影响真实危险提示。

## 10. 推荐首轮 Sprint

如果按 1 到 2 周一个 Sprint 推进，首轮建议只做 P0 + M1 骨架，不碰联网和复杂亡者。

### Sprint 1：能跑起来

任务：

- T0.2 建立 Web 项目骨架。
- T0.3 制定 UI token 与 HUD 线框。
- T1.1 实现场景与渲染基础。
- T1.2 实现输入系统。
- T1.3 实现移动与翻滚。

验收：

- 浏览器能打开原型。
- 一个角色能移动、瞄准、翻滚。
- 基础 HUD 不遮挡画面。

### Sprint 2：能互相杀

任务：

- T1.4 实现蓄力射箭与假动作。
- T1.5 实现基础肢体状态。
- T1.6 实现基础追击链。
- T1.8 实现基础 HUD。

验收：

- 可完成基础追击击杀。
- 断臂和断腿影响操作。
- 玩家知道自己要追谁。

### Sprint 3：像一局游戏

任务：

- T1.7 实现地图交互元素。
- T1.9 实现本地结算页。
- T1.10 M1 原型验收。
- Q1 增加规则测试。
- Q2 做性能预算初版。

验收：

- 可完成一局本地原型。
- 有地图策略、有结算、有重开。

## 11. 当前不做的任务

以下内容暂缓，避免首版范围失控：

- 完整排位系统。
- 复杂账号系统。
- 真实付费抽卡。
- 多地图池。
- 移动端完整对战操作。
- 高级 AI。
- 复杂反作弊平台接入。
- WebRTC 低延迟重构。

## 12. 任务完成定义

每个任务完成前至少满足：

1. 功能按任务验收标准可操作。
2. 关键状态有可视反馈。
3. 配置、类型或文档同步更新。
4. 已运行与任务相关的最小验证。
5. 若无法验证，必须记录原因和风险。
