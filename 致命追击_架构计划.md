# 《致命追击：混沌法则》架构计划

## 1. 架构目标

本项目的核心技术目标是支撑一款高混乱度、强反馈、多人公平竞技游戏。架构必须优先保证以下能力：

1. 局内规则公平：追击链、击杀归属、复活夺舍、空投技能等关键逻辑由权威端判定。
2. 操作响应快速：移动、瞄准、蓄力、翻滚、弹反等高频操作需要低延迟反馈。
3. 混乱画面可读：几何风格、美术特效、断肢血浆和布娃娃表现必须服务于战斗信息识别。
4. 断肢影响实战：肢体损失不是纯表现，需要进入移动、瞄准、近战、寄生和结算规则。
5. 死亡后仍可玩：亡者机制不是旁支，而是与生者战斗闭环同级的核心系统。
6. 易于扩展内容：皮肤、喷涂、箭矢拖尾、断肢特效、地图组件和技能道具需要数据驱动配置。

## 2. 推荐技术路线

### 2.1 网页端技术栈

目标平台调整为网页端，首版推荐采用 TypeScript + Canvas/WebGL 的浏览器游戏架构。

推荐组合：

- 渲染：PixiJS 或 Phaser 3。若更重视工程可控和自定义表现，优先 PixiJS；若更重视快速原型，优先 Phaser 3。
- 物理：Matter.js 负责轻量刚体、碰撞、木箱破坏和本地布娃娃表现。
- 客户端工程：TypeScript + Vite。
- 服务端：Node.js + TypeScript。
- 通信：WebSocket，后续如需更低延迟可评估 WebRTC DataChannel。

首版建议优先 PixiJS + Matter.js + Node.js。原因是本项目需要大量自定义 UI、假射线、箭矢拖尾、血浆粒子和断肢表现，PixiJS 的渲染控制更灵活；规则和判定仍由 Node.js 权威服务端负责。

### 2.2 网络架构

推荐使用权威服务器模型。

- 服务端负责：玩家状态、追击链、命中判定、击杀归属、亡者形态、空投技能、毒圈、结算。
- 客户端负责：输入采集、预测表现、本地动画、粒子、音效、UI、非权威布娃娃表现。
- 同步策略：核心战斗状态快照同步，表现型碎尸、血浆、涂鸦、拖尾按事件同步或本地派生。

不建议首版使用纯 P2P。追击链计分、夺舍复活和空手接白刃都容易出现争议，需要服务端统一裁决。

## 3. 分层架构

```text
Client
├─ Input Layer          输入、按键缓冲、取消动作、轮盘选择
├─ Presentation Layer   PixiJS 渲染、特效、音效、UI、镜头、布娃娃表现
├─ Prediction Layer     本地移动预测、射击预表现、回滚修正
├─ Game Proxy           服务端状态镜像、事件播放、实体插值
└─ Asset Runtime        皮肤、喷涂、拖尾、断肢特效装配

Authoritative Server
├─ Match Orchestrator   房间、开局、阶段、毒圈、结算
├─ Rule Engine          追击链、得分、惩罚、目标转移
├─ Combat Engine        蓄力射击、弹反、翻滚无敌帧、命中裁决
├─ Limb Damage Engine   肢体损失、能力削弱、濒死规则、断肢归属
├─ Afterlife Engine     寄生肉盾、幽灵老六、夺舍复活
├─ Physics Arbiter      碰撞、掩体破坏、大招关键判定
├─ Map Runtime          随机地图、草丛、泥沼、黑墙、木箱
└─ Economy Gateway      对局奖励、外观解锁、反作弊校验

Shared
├─ Config Schema        技能、地图、道具、角色、结算奖项配置
├─ Protocol             输入包、快照包、事件包、结算包
└─ Simulation Types     实体状态、肢体状态、伤害事件、击杀事件、亡者状态
```

## 4. 核心模块设计

### 4.1 输入与动作模块

输入模块需要将玩家操作抽象为可同步的意图，而不是直接同步坐标。

关键动作：

- 移动：WASD 方向向量。
- 瞄准：鼠标世界坐标或角度。
- 蓄力射击：开始蓄力、取消蓄力、释放射击。
- 翻滚：消耗冷却并触发 0.5 秒无敌帧。
- 弹反：短窗口判定，服务端确认是否成功。
- 社交：表情轮盘、喷涂放置。

输入包建议包含 tick、移动方向、瞄准角度、动作位掩码和客户端序号，用于预测与校正。

### 4.2 战斗模块

战斗模块负责所有生者之间的有效对抗。

首版需要支持：

1. 弓箭蓄力：蓄力时间决定速度、伤害范围或穿透能力。
2. 假动作：取消蓄力只产生音效和姿态表现，不生成投射物。
3. 投射物：箭矢飞行、命中、反弹、擦弹事件。
4. 翻滚：服务端记录无敌帧窗口，客户端提前播放。
5. 弹反/拼刀：短时间窗口内对箭矢或近战攻击进行方向性判定。

命中结果必须由服务端确认。客户端可先播放预表现，但需要允许被服务端回滚或修正。

### 4.3 肢体损失模块

肢体损失是局内实战规则，不是单纯血腥表现。建议独立成 `LimbDamageService`，由服务端权威维护每个生者的肢体状态。

基础肢体：

- 头部：命中通常直接死亡，特殊技能或护盾状态可例外。
- 躯干：承载生命状态，严重受损会进入濒死或直接死亡。
- 左臂：影响蓄力稳定、弹反窗口和双手武器操作。
- 右臂：影响拉弓速度、射击精度和近战拼刀。
- 左腿：影响移动速度、转向、翻滚距离。
- 右腿：影响冲刺、翻滚冷却恢复和泥沼脱离能力。

建议首版断肢效果：

```text
LoseArm
├─ chargeSpeedMultiplier = 0.65
├─ aimShake += medium
└─ parryWindowMultiplier = 0.6

LoseBothArms
├─ cannotUseBow = true
├─ canUseBiteOrKick = true
└─ canStillTriggerGhostOrParasiteAfterDeath = true

LoseLeg
├─ moveSpeedMultiplier = 0.6
├─ rollDistanceMultiplier = 0.5
└─ mudSlowExtra = true

LoseBothLegs
├─ crawlOnly = true
├─ cannotRoll = true
└─ grassStealthBonus = true
```

断肢来源：

- 箭矢精准命中。
- 几何爆裂箭反弹命中。
- 流星锤砸击。
- 空间折叠碾压。
- 木箱爆裂或地图挤压。

裁决原则：

- 服务端根据命中部位、攻击类型、速度、蓄力值和目标当前状态决定断肢。
- 客户端只播放断肢表现，不能自行声明肢体损失。
- 肢体状态必须进入快照同步，因为它会改变操作能力。
- 断肢应有短暂强反馈，但 UI 提示要克制，避免遮挡战斗信息。

### 4.4 追击链规则模块

追击链是本项目最重要的规则模块，建议独立成 `HuntChainService`。

职责：

- 开局生成单向闭环目标链。
- 维护每名玩家的当前目标。
- 判定击杀是否有效得分。
- 处理目标被第三方击杀后的惩罚和目标转移。
- 向客户端同步弱提示罗盘，不暴露完整链路。

关键事件：

```text
PlayerKilled(killer, victim, cause)
├─ victim == killer.currentTarget
│  ├─ killer.score += 1
│  └─ killer.currentTarget = victim.currentTarget or next valid target
└─ victim != killer.currentTarget
   ├─ victim.hunter receives exposure penalty
   └─ victim.hunter.currentTarget = killer
```

边界情况需要在首版设计清楚：

- 环境击杀归属。
- 毒圈击杀归属。
- 同一 tick 多人死亡。
- 目标已死亡但亡者仍可夺舍。
- 只剩两名或一名生者时的链路收敛。

### 4.5 亡者系统模块

亡者系统建议作为与生者状态机并列的 `AfterlifeStateMachine`。

死亡后状态：

```text
Alive
└─ DeadSelecting
   ├─ ParasiteShield
   └─ GhostTrickster
```

寄生肉盾：

- 绑定宿主通常为击杀者。
- 作为宿主身上的独立挂点实体存在。
- 可触发擦箭检测和空手接白刃。
- 成功后产生宿主死亡、亡者夺舍、身份转换事件。
- 如果宿主缺臂，寄生小弟的空手接白刃判定更容易触发，但夺舍后的新身体会继承当前肢体损伤。

幽灵老六：

- 默认隐形、不可碰撞、不可受伤。
- 可发出假拉弓音效和红色瞄准射线。
- 释放假射线时显形 0.5 秒。
- 幽灵行为应消耗冷却，防止无限噪声污染。

### 4.6 地图与环境模块

地图使用房间模板加组件化地形生成，避免完全随机导致不可控。

地图元素：

- 合金黑墙：不可破坏，可反弹特定箭矢。
- 木箱掩体：有生命值，可被箭矢或流星锤破坏。
- 高草丛：进入后隐身，离开后短暂残影。
- 泥沼区：降低移动速度和翻滚距离。
- 空投点：按阶段刷新高风险高收益道具。

地图数据建议拆为：

- 静态布局：墙体、出生点、草丛、泥沼区。
- 动态对象：木箱、空投、可破坏物。
- 规则区域：毒圈、安全区、视野惩罚区。

### 4.7 大招技能模块

特殊技能建议采用数据驱动的 `SkillDefinition + SkillRuntime`。

空间折叠：

- 步骤：射出锚点、确认挂墙、积累张力、释放回弹、路径碾压判定。
- 服务端判定锚点合法性、张力上限、碾压路径和击杀结果。
- 客户端负责墙体拉伸表现、音效、震屏和碎尸特效。
- 碾压可造成按路径接触部位的断肢，中心区域命中才直接秒杀。

人肉流星锤：

- 宿主与寄生小弟构成链式物理表现。
- 服务端只裁决关键碰撞和伤害，不同步每个布娃娃骨骼。
- 小弟的千斤坠输入影响惯性参数，需要服务端限幅。
- 砸击优先造成断腿、断臂和击飞，蓄满速度或撞墙夹击时才直接致死。

### 4.8 表现与血腥反馈模块

血腥反馈应分为规则事件、肢体状态和本地表现。肢体状态会影响玩法，必须由服务端同步；血浆、碎肉和布娃娃则尽量本地派生。

规则事件：

- 死亡。
- 断肢类型和命中部位。
- 击飞方向。
- 尸体归属。
- 结算统计。

肢体状态：

- 当前剩余肢体。
- 肢体损伤等级。
- 由断肢造成的能力修正。
- 临时止血、硬直、濒死或爬行状态。

本地表现：

- 血浆粒子。
- 断肢飞散。
- 布娃娃摆动。
- 屏幕震动。
- 音效混音。

这样可以减少网络同步压力，也能让低配置客户端降低特效质量而不影响竞技公平。

### 4.9 局外成长与经济模块

局外成长必须只影响外观。

系统包括：

- 碎肉代币。
- 皮肤。
- 断肢特效。
- 箭矢拖尾。
- 喷涂。
- 表情包。

服务端需要校验玩家拥有的外观资源，客户端只负责展示。所有对局内判定不得读取外观配置。

## 6. 服务端状态模型

核心实体：

- `PlayerSession`：连接、账号、输入序号、延迟信息。
- `Combatant`：生者状态、血量或一击必杀状态、位置、速度、目标、肢体状态。
- `LimbState`：头、躯干、左臂、右臂、左腿、右腿的完整、受损、断裂状态。
- `AfterlifeEntity`：亡者形态、宿主、冷却、显形窗口。
- `Projectile`：箭矢位置、速度、反弹次数、所有者、蓄力值。
- `MapObject`：木箱、空投、可破坏物、区域效果。
- `MatchState`：阶段、剩余时间、毒圈、追击链、排行榜。

推荐 tick 频率：

- 服务端模拟：30 或 60 tick。
- 输入上报：每 tick 或合并上报。
- 快照下发：10 到 20 次/秒，客户端插值。

## 7. 反作弊与公平性

首版必须保护以下判定：

1. 移动速度、翻滚冷却、无敌帧时间。
2. 蓄力时长、取消射击、释放时间。
3. 箭矢命中、擦弹、反弹。
4. 肢体命中、断肢结果、由肢体损失产生的能力削弱。
5. 草丛隐身和暴露视野。
6. 亡者夺舍和幽灵显形窗口。
7. 外观资源拥有权。

客户端不得直接声明击杀成功，只能提交输入与本地预测事件。

## 8. 开发里程碑

### M0：纸面原型与规则验证

目标：验证追击链、目标转移、亡者干扰是否形成乐趣闭环。

产出：

- 规则状态机。
- 追击链事件表。
- 首版数值表。
- 最小地图草图。

### M1：本地单机战斗原型

目标：让 2 到 4 个本地实体完成基础战斗。

范围：

- WASD 移动。
- 鼠标瞄准。
- 蓄力射箭。
- 翻滚无敌帧。
- 基础肢体损失和能力削弱。
- 木箱、草丛、泥沼。
- 本地击杀与追击链更新。

验收：

- 玩家能理解自己该追谁。
- 假动作能骗出翻滚或弹反。
- 草丛和掩体能制造伏击。

### M2：亡者机制原型

目标：死亡后仍有强参与感。

范围：

- 死亡形态选择。
- 寄生肉盾。
- 空手接白刃。
- 幽灵假射线。
- 显形窗口。
- 夺舍后继承或重置肢体损伤的规则验证。

验收：

- 被杀玩家 10 秒内有事可做。
- 亡者能影响战局但不能无成本统治战局。
- 生者能通过观察反制幽灵。

### M3：权威联网原型

目标：验证网络同步和核心公平性。

范围：

- 房间与匹配。
- 输入同步。
- 服务端移动与命中裁决。
- 客户端预测和插值。
- 追击链服务端权威。

验收：

- 4 到 8 人对局可玩。
- 命中争议可解释。
- 高延迟下仍能完成基本操作。

### M4：混沌中盘与大招

目标：加入空投技能和毒圈，形成局势升级。

范围：

- 空投刷新。
- 空间折叠。
- 人肉流星锤。
- 毒圈缩小。
- 结算奖项。

验收：

- 中后期有明显戏剧性。
- 大招强但可预判、可躲避。
- 结算能强化沙雕叙事。

### M5：内容与商业化外观

目标：建立可持续内容生产管线。

范围：

- 碎肉代币。
- 外观背包。
- 皮肤、喷涂、拖尾、断肢特效。
- 抽取或兑换系统。

验收：

- 外观不影响局内判定。
- 新增一个外观不需要改战斗代码。
- 对局奖励与结算统计闭环。

## 9. 风险与应对

### 9.1 物理同步风险

风险：布娃娃、流星锤、墙体回弹如果全量同步，会导致网络成本高且表现不稳定。

应对：服务端只同步关键判定，客户端本地生成夸张表现。

### 9.2 信息噪声风险

风险：幽灵假射线、血浆、断肢、喷涂太多会遮挡战斗信息。

应对：建立视觉优先级。真实箭矢、目标罗盘、危险技能预警始终高于装饰特效。

### 9.3 规则边界风险

风险：第三方击杀、环境击杀、夺舍复活容易让玩家觉得不公平。

应对：所有特殊裁决都要有明确事件提示和结算解释。

### 9.4 亡者过强风险

风险：死亡玩家如果影响过大，会让生者挫败。

应对：亡者技能设置冷却、显形、资源消耗和反制窗口。

## 10. 首版最小可玩范围

首个可玩版本建议只做以下内容：

- 4 人房间。
- 一个小型竞技场。
- 弓箭蓄力、取消、翻滚。
- 断臂影响射击、断腿影响移动。
- 追击链和目标转移。
- 木箱、草丛、泥沼。
- 寄生肉盾和幽灵假射线。
- 简化血浆、断肢表现和肢体状态 UI。
- 基础结算奖项。

暂缓内容：

- 完整商业化。
- 大量皮肤。
- 复杂抽卡。
- 多地图池。
- 排位系统。
- 复杂 AI。

## 11. 建议目录结构

